# Agent流式执行Trace可视化系统 需求文档

文档版本：V1.0
编制日期：2026年04月
产品名称：Agent流式执行Trace可视化系统（简称：StreamTrace）
对标产品：Anthropic Claude Code、开源OpenCode



## 一、文档概述

### 1.1 文档目的

本文档完整定义StreamTrace系统的业务需求、功能边界、数据规范、技术要求与验收标准，作为项目POC验证、MVP开发、迭代优化的唯一需求依据，确保团队对需求的理解一致，同时适配开源项目的标准化规划。

### 1.2 术语定义

| 术语 | 标准定义 |
|------|----------|
| Claude Code | Anthropic官方推出的Agent全链路可视化界面，核心能力为Agent执行过程的流式树形节点Trace可视化 |
| OpenCode | 开源社区对标Claude Code的Agent可视化方案，核心为可定制的流式树形渲染框架 |
| SSE | Server-Sent Events，服务端推送事件，本系统唯一的流式数据传输协议 |
| Agent | 公司内部各类AI智能体，本系统的流式数据生产方，不可控，仅能约定其输出数据协议 |
| Trace节点 | Agent执行过程中产生的最小可视化单元，包括用户输入、助手思考、工具调用、代码执行、执行结果等类型 |
| 流式渲染 | 基于SSE推送的增量数据，逐段更新节点内容，仅重绘变化节点，无全量页面刷新的渲染方式 |
| MongoDB | 公司内部流数据持久化数据库，不可控，仅能兼容其推送的合规数据 |

### 1.3 读者对象

产品经理、前端开发工程师、后端开发工程师、测试工程师、开源项目维护者、Agent接入方开发人员。


## 二、需求背景与业务目标

### 2.1 业务背景

公司内部存在大量独立开发的Agent智能体，所有Agent均通过SSE协议流式输出执行过程数据；当前存在以下核心痛点：

1. 各Agent独立开发可视化界面，重复造轮子，开发成本高，体验不统一；
2. 现有方案无法实现Agent执行全链路的树形Trace可视化，无法还原Agent的思考、调用、执行完整流程；
3. 流式渲染性能差，高频SSE数据推送易导致页面全量重绘、卡顿；
4. 无法兼容多前端技术栈，Vue/React项目需重复开发适配；
5. Agent端与MongoDB端均已成型，无法进行内部逻辑改造，仅能通过约定数据协议实现接入。

### 2.2 核心目标

1. 统一体验：打造对标Claude Code的标准化Agent执行Trace可视化方案，实现全公司Agent流式输出的统一渲染体验；
2. 极简接入：仅通过约定SSE数据协议实现Agent接入，无需改造Agent与MongoDB内部逻辑，接入成本为0；
3. 高性能渲染：实现流式增量渲染，仅更新变化节点，支持高频SSE数据推送无卡顿，无限层级树形结构流畅展示；
4. 跨栈兼容：核心逻辑框架无关，官方提供Vue3、React18适配包，可无缝接入不同技术栈的前端项目；
5. 开源可扩展：打造可开源的标准化脚手架，支持自定义节点样式、交互行为、功能插件，满足不同业务场景的定制化需求。

### 2.3 对标产品核心能力对齐

| 对标能力 | Claude Code 原生能力 | 本系统对齐要求 |
|----------|---------------------|----------------|
| 流式逐字输出 | 支持Agent内容逐字流式渲染，无卡顿 | 必须对齐，POC阶段实现 |
| 树形Trace可视化 | 无限层级树形结构，还原Agent执行全链路 | 必须对齐，POC阶段实现 |
| 节点类型差异化渲染 | 按思考/工具调用/代码执行等类型展示不同样式 | 必须对齐，POC阶段实现 |
| 节点状态实时展示 | 流式输出中/完成/失败状态实时同步 | 必须对齐，POC阶段实现 |
| 执行链路回溯 | 支持查看完整执行历史，节点展开/折叠 | MVP阶段实现 |
| 跨框架兼容 | 仅支持官方Web端 | 本系统超越，支持Vue3/React18双框架 |
| 自定义扩展 | 仅支持官方固定能力 | 本系统超越，支持全量自定义扩展 |


## 三、范围与约束

### 3.1 需求范围

#### 3.1.1 必须实现范围

1. 定义全链路统一的SSE流式数据协议，作为Agent接入的唯一标准；
2. 实现SSE流的接入、解析、校验、重连、异常处理全流程管理；
3. 实现流式节点数据的状态管理、树形结构自动构建、增量更新；
4. 实现对标Claude Code的流式增量渲染，支持无限层级树形结构展示；
5. 实现节点类型差异化样式、状态实时展示、基础交互能力；
6. 提供Vue3、React18官方适配包，核心逻辑框架无关。

#### 3.1.2 后续迭代范围

1. 节点展开/折叠、复制、编辑、重试、收藏等高级交互；
2. 执行链路回溯、历史版本查看、搜索筛选能力；
3. 代码高亮、Markdown渲染、图表展示、图片渲染等富媒体节点支持；
4. 虚拟滚动，支持1000+节点的流畅渲染；
5. 离线缓存、断点续传、数据导出能力；
6. 插件化扩展体系，支持自定义节点、自定义交互、第三方能力集成。

### 3.2 核心约束边界（不可突破）

1. Agent端不可控：不要求Agent端做任何内部逻辑改造，仅需其按约定的SSE协议推送流式数据，不得强制Agent新增非必要能力；
2. MongoDB端不可控：不涉及MongoDB的开发、改造、运维，仅兼容其完成数据存储后推送的、符合协议规范的SSE数据，不得要求MongoDB修改存储结构；
3. 后端仅做透传：后端服务仅负责SSE流的转发、协议合规性校验，不做业务逻辑处理、数据存储、格式转换，不得新增强依赖后端的业务能力；
4. 前端核心闭环：所有核心业务逻辑、状态管理、渲染逻辑均在前端实现，不依赖后端服务的特殊能力，保证前端可独立运行、离线兼容。

## 五、核心功能需求

### 5.1 SSE流接入与管理模块

#### 5.1.1 SSE连接管理

1. 支持通过配置SSE服务地址，一键初始化连接，兼容HTTP/HTTPS协议；
2. 支持自定义请求头（如鉴权Token、租户ID），满足企业内部鉴权需求；
3. 连接状态实时同步，支持 连接中/已连接/已断开/连接失败 4种状态展示；
4. 支持手动断开、重连、重置连接，重置时清空本地所有节点数据。

#### 5.1.2 消息处理能力

1. 自动解析SSE消息体，严格校验数据协议合规性，不合规消息丢弃并打印告警日志；
2. 支持消息去重，基于消息唯一ID过滤重复推送的消息，避免节点重复渲染；
3. 支持消息防乱序，基于消息序列号保证节点内容按正确顺序拼接；
4. 支持分片消息自动拼接，大内容分片推送时，自动合并为完整节点内容。

#### 5.1.3 异常与重连机制

1. 连接异常断开时，自动触发指数退避重连，重连间隔为1s/2s/4s/8s，最大间隔30s；
2. 网络从离线恢复为在线时，自动触发重连，无需手动操作；
3. 重连过程中保留已渲染的节点数据，不清空页面，保证用户体验；
4. 消息解析失败、协议校验失败时，控制台输出详细错误日志，不影响主线程运行。

### 5.2 节点数据状态管理模块

1. 采用扁平化Map结构存储节点数据，以 nodeId 为唯一主键，支持O(1)复杂度的节点查找、更新；
2. 支持2种核心事件处理：
   - node_create：创建新节点，存入Map，初始化节点内容、状态、元数据；
   - node_append：精准定位目标节点，追加内容片段，更新节点状态；
3. 采用不可变数据更新模式，仅修改目标节点数据，不改动其他节点，保证增量渲染的稳定性；
4. 节点状态全生命周期管理，支持 streaming（流式输出中）/complete（执行完成）/error（执行失败） 3种核心状态；
5. 支持节点元数据存储，包括创建时间、更新时间、节点类型、所属AgentID等，不修改核心数据结构。

### 5.3 树形结构构建模块

1. 基于 parentId 自动构建无限层级树形结构，根节点 parentId 为 null ，支持无限深度的子节点嵌套；
2. 支持实时增量更新树形结构，新增/修改节点时，仅更新对应分支，不重构整棵树；
3. 支持深度优先、广度优先两种树形遍历方式，提供节点查找、子节点获取、层级计算等工具方法；
4. 支持异常节点兼容，父节点未创建时，子节点暂存等待，父节点创建后自动挂载，不报错、不丢失数据。

### 5.4 流式增量渲染模块

1. 核心增量渲染能力：仅当节点内容、状态发生变化时，才重绘对应节点，其他节点完全不刷新，杜绝全量页面重绘；
2. 逐字流式输出：支持SSE推送的内容片段逐字渲染，还原Claude Code的流式输出效果，无跳字、卡顿；
3. 渲染调度优化：高频SSE推送时，自动触发批量渲染防抖（默认50ms，可配置），避免浏览器主线程阻塞；
4. 递归树形渲染：支持无限层级树形节点的递归渲染，子节点自动按层级缩进，与Claude Code视觉效果一致；
5. 渲染性能兜底：节点数量超过阈值时，自动触发渲染降级，保证页面流畅度不下降。

### 5.5 可视化交互模块

#### 5.5.1 基础交互（POC阶段实现）

1. 节点类型差异化样式渲染，不同类型节点展示不同的背景色、边框、标签样式，与Claude Code对齐；
2. 节点状态实时展示， streaming 状态节点展示加载动画， error 状态节点展示错误提示；
3. 层级自动缩进，子节点按层级深度自动向左缩进，清晰展示树形链路关系；
4. 长内容自动换行，支持代码块、长文本的自动换行，不出现横向滚动条。

#### 5.5.2 高级交互（MVP阶段实现）

1. 节点展开/折叠，支持子节点的展开与隐藏，默认全部展开；
2. 节点内容复制，一键复制节点完整内容；
3. 执行链路回溯，支持查看节点的完整创建、更新历史；
4. 节点搜索筛选，支持按节点类型、状态、关键词搜索定位节点；
5. 失败节点重试，支持对 error 状态的节点触发重试操作。


## 六、核心数据协议规范（强约束，全链路必须遵守）

### 6.1 SSE通信标准规范

1. 遵循HTTP原生SSE协议标准，单条消息的固定格式为：

```
data: {JSON字符串}\n\n
```

2. 字符编码统一为UTF-8，响应头必须包含以下字段：

```
Content-Type: text/event-stream; charset=utf-8
Cache-Control: no-cache
Connection: keep-alive
Access-Control-Allow-Origin: *
```

3. 消息推送必须为单条串行推送，不得批量打包多条消息，保证流式渲染的实时性。

### 6.2 SSE流式消息核心结构

单条SSE消息的JSON体必须符合以下结构，无特殊说明均为必填字段：

```typescript
interface SSEStreamMessage {
  /**
   * 全局唯一消息ID，UUID格式，用于消息去重、丢失追溯、重传
   * 必填
   */
  msgId: string;
  /**
   * 事件类型，仅支持2种核心类型，覆盖全量场景
   * 必填：node_create（新建节点）/node_append（追加节点内容/更新状态）
   */
  type: 'node_create' | 'node_append';
  /**
   * 节点数据体，不同事件类型对应不同的必填字段
   * 必填
   */
  data: StreamNode;
  /**
   * 消息序列号，自增数字，用于防乱序、检测消息丢失
   * 必填
   */
  seq: number;
  /**
   * 消息推送时间戳，13位毫秒级
   * 必填
   */
  timestamp: number;
  /**
   * 协议版本号，语义化版本，用于向后兼容
   * 选填，默认v1.0.0
   */
  protocolVersion?: string;
}
```

### 6.3 流式节点数据结构规范

StreamNode 为节点核心数据结构，全链路统一，不同事件类型的必填字段如下：

```typescript
interface StreamNode {
  /**
   * 全局唯一节点ID，Agent生成，UUID/自增ID均可，前端用于定位节点
   * 【全事件必填】node_create/node_append均必须传递
   */
  nodeId: string;
  /**
   * 父节点ID，根节点传null，用于构建树形层级
   * 【node_create必填】，node_append选填
   */
  parentId?: string | null;
  /**
   * 节点类型，固定枚举值，前端用于差异化样式渲染
   * 【node_create必填】，node_append选填
   */
  nodeType?: 'user_input' | 'assistant_thought' | 'tool_call' | 'code_execution' | 'execution_result' | 'final_output';
  /**
   * 本次推送的内容片段
   * node_create：传节点初始内容
   * node_append：传本次追加的文本片段，无内容传空字符串
   * 【全事件必填】
   */
  chunk: string;
  /**
   * 节点状态，默认streaming
   * 选填：streaming（流式输出中）/complete（执行完成）/error（执行失败）
   */
  status?: 'streaming' | 'complete' | 'error';
  /**
   * 所属Agent唯一标识，用于多Agent数据隔离
   * 选填
   */
  agentId?: string;
  /**
   * 自定义扩展字段，必须以x-为前缀，不得与核心字段重名
   * 选填
   */
  [key: `x-${string}`]: any;
}
```

### 6.4 事件类型与处理规则

| 事件类型 | 使用规则 | 前端处理逻辑 |
|----------|----------|--------------|
| node_create | 1. 新建节点时必须使用此事件；2. 必须传递所有必填字段；3. 同一nodeId仅能推送一次，重复推送视为无效消息 | 1. 校验nodeId是否已存在，不存在则创建新节点存入Map；2. 触发树形结构更新；3. 渲染新增节点 |
| node_append | 1. 节点内容追加、状态更新时必须使用此事件；2. 必须传递nodeId+chunk，其他字段选填；3. 节点结束时，chunk传空字符串，status传complete | 1. 定位目标节点，拼接chunk到完整内容；2. 更新节点状态；3. 仅触发对应节点的重绘，实现增量渲染 |


## 七、接口规范

### 7.1 SSE推送接口规范

| 项目 | 规范要求 |
|------|----------|
| 请求方式 | GET |
| 接口路径 | 可配置，默认 /api/agent/stream |
| 请求头 | 支持自定义鉴权头、租户头，默认携带 Accept: text/event-stream |
| 请求参数 | 支持URL Query参数，如agentId、requestId、traceId，用于过滤指定Agent的流数据 |
| 响应格式 | 严格遵循6.1节SSE通信标准规范 |
| 字符编码 | UTF-8 |

### 7.2 错误码规范

SSE推送过程中，异常场景需通过专用错误事件推送，格式如下：

```
event: error
data: {"code": "ERROR_CODE", "message": "错误描述", "timestamp": 1234567890}\n\n
```

核心错误码定义：

| 错误码 | 错误描述 | 前端处理逻辑 |
|--------|----------|--------------|
| 400 | 协议格式错误 | 控制台打印错误，不中断连接 |
| 401 | 鉴权失败 | 断开连接，展示鉴权失败提示 |
| 404 | 流资源不存在 | 断开连接，展示资源不存在提示 |
| 500 | 服务端内部错误 | 触发自动重连，打印错误日志 |


## 八、前端渲染规范

### 8.1 节点类型与样式规范（对齐Claude Code）

| 节点类型 | 样式规范 | 应用场景 |
|----------|----------|----------|
| user_input | 浅灰色背景，灰色边框，标签为「用户输入」 | 用户输入的提问、指令 |
| assistant_thought | 白色背景，紫色边框，标签为「助手思考」，紫色标签文字 | Agent的思考过程、逻辑推理 |
| tool_call | 浅蓝色背景，蓝色边框，标签为「工具调用」，蓝色标签文字 | Agent调用工具、API的请求内容 |
| code_execution | 浅紫色背景，紫色边框，标签为「代码生成」，紫色标签文字 | Agent生成的代码、脚本内容 |
| execution_result | 浅绿色背景，绿色边框，标签为「执行结果」，绿色标签文字 | 工具、代码的执行返回结果 |
| final_output | 白色背景，深紫色主边框，标签为「最终输出」，深紫色标签文字 | Agent给用户的最终回复 |
| error | 浅红色背景，红色边框，标签为「执行失败」，红色标签文字 | 执行失败、报错的节点 |

### 8.2 流式渲染规则

1. 节点内容必须按SSE推送的顺序逐段渲染，不得乱序、跳字；
2. 仅当节点的 content 或 status 发生变化时，才重绘对应节点，其他节点保持不变；
3. 流式输出中的节点，必须展示加载动画，状态变为 complete 后，动画自动消失；
4. 子节点必须在父节点渲染完成后挂载，层级缩进与父节点保持固定间距（默认32px/层级）；
5. 代码块、换行符、空格必须保留原始格式，不得自动格式化、压缩。

### 8.3 交互规范

1. 节点hover时，展示轻微的阴影效果，提升交互反馈；
2. 长内容节点默认展示完整内容，超过300px高度时，自动添加纵向滚动条；
3. 点击节点标签，可快速定位到对应节点的父节点/子节点；
4. 页面滚动时，保持当前流式输出的节点在可视区域内。


## 九、非功能需求

### 9.1 性能需求

1. 渲染性能：单条SSE消息的渲染耗时≤10ms，高频推送（每秒10条）时，页面帧率稳定在60fps，无卡顿、掉帧；
2. 增量渲染：新增/更新节点时，仅重绘目标节点，DOM操作量减少90%以上；
3. 内存占用：节点数量≤1000个时，浏览器内存占用≤200MB，无内存泄漏；
4. 首屏加载：核心包gzip后体积≤50KB，首屏加载时间≤1s；
5. 兼容性：支持同时接入≥10个Agent的并行SSE流，无冲突、无数据错乱。

### 9.2 兼容性需求

1. 浏览器兼容：支持Chrome 90+、Firefox 88+、Edge 90+、Safari 14+，覆盖95%以上的主流浏览器；
2. 框架兼容：核心层无框架依赖，官方支持Vue 3.2+、React 18+，提供开箱即用的适配包；
3. 协议兼容：向后兼容所有历史版本的协议，协议升级不影响已接入的Agent；
4. 移动端兼容：支持移动端浏览器自适应，样式无错乱，交互流畅。

### 9.3 可用性需求

1. 系统可用性：全年可用率≥99.9%，非计划停机时间≤8.76小时/年；
2. 异常兜底：SSE连接异常、消息解析失败时，系统不崩溃、不白屏，保留已渲染数据；
3. 接入成本：Agent接入仅需遵守数据协议，无需额外开发，接入时间≤10分钟；
4. 文档完整性：提供完整的接入文档、API文档、示例代码，覆盖所有核心场景。

### 9.4 可扩展性需求

1. 插件化扩展：支持通过插件机制扩展自定义节点、自定义渲染、自定义交互，不侵入核心代码；
2. 样式自定义：支持全量自定义节点样式、布局、主题，无需修改核心渲染逻辑；
3. 节点扩展：支持新增自定义节点类型，无需修改核心状态管理、树形构建逻辑；
4. 框架扩展：可快速适配Angular、Svelte等其他前端框架，核心逻辑完全复用。

### 9.5 可维护性需求

1. 代码规范：严格遵循TypeScript类型规范，核心代码注释覆盖率≥90%；
2. 测试覆盖：核心代码单元测试覆盖率≥90%，集成测试覆盖率≥80%；
3. 版本管理：遵循语义化版本（SemVer）规范，主版本号做不兼容变更，次版本号做向下兼容的新特性，补丁版本号做bug修复；
4. 日志规范：提供分级日志系统，支持debug/info/warn/error四级日志，便于问题排查。

|----------|----------|----------|
| Agent端不遵守数据协议，推送不合规数据 | 中 | 1. 前端做严格的协议校验，不合规数据直接丢弃，不影响主线程；2. 提供详细的接入文档、示例代码、校验工具，降低Agent接入成本；3. 提供协议校验SDK，Agent端可自行校验数据合规性 |
